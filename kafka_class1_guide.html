02 What is Apache Kafka?

Apache Kafka is a distributed event streaming platform. Think of it as a very fast, very durable, ordered log of messages. Applications write to it (producers), and other applications read from it (consumers) — independently, at their own pace.

📖 The real-world analogy: a newspaper printing press

A newspaper printing press prints thousands of copies of the same edition. Readers (consumers) can buy a copy at any time — the press doesn't need to know who will read it or when. If you miss today's paper, the archive still has it.

Kafka works the same way: the broker is the press, the topic is the edition name, the producer is the journalist filing the story, and the consumers are readers — each reading at their own pace, independently.

This decoupling is the key insight: the producer doesn't care whether there are 0 or 100 consumers. The consumer doesn't care how fast the producer writes.

📖 Why not just write directly to the database?

In a simple system, the producer could write directly to PostgreSQL. That works fine at low volume. The problems appear at scale:

Backpressure: If the database is slow or down, the producer has nowhere to send data — you lose orders.

Coupling: Every service that needs the data must connect to the same database. Schema changes break everything simultaneously.

No replay: If a consumer has a bug and processes data incorrectly, you can't re-run it. Database inserts are final.

Kafka solves all three. It acts as a durable buffer — producers write fast, consumers read at their own pace, and messages can be replayed any number of times from the beginning of the topic.

📖 What is ZooKeeper and why does Kafka need it?

ZooKeeper is a separate coordination service that Kafka uses to manage its own cluster metadata.

Kafka needs to track: which brokers are alive, which broker is the "leader" for each partition, and where consumer groups left off. ZooKeeper stores all of that state in a reliable, distributed way.

Think of ZooKeeper as the cluster's directory service. You don't interact with it directly — Kafka does, automatically, in the background.

Note: Modern Kafka (v3.3+, KRaft mode) no longer requires ZooKeeper. But the vast majority of production clusters and tutorials still use it, so we include it here to build the full mental model. In our docker-compose.yml you'll see both services side-by-side.

📖 What is Docker Compose doing here?

Docker packages software and all its dependencies into an isolated container — like a lightweight virtual machine that runs the same on any laptop. No need to install Kafka or PostgreSQL directly on your computer.

Docker Compose lets you define and run multiple containers together with a single YAML file. In our case:

• zookeeper — starts first, listens on internal port 2181.
• kafka — starts after zookeeper (via depends_on), listens on port 9092.
• postgres — fully independent, listens on port 5432.

docker-compose up -d starts all three in the background. docker-compose down stops and removes them. docker ps shows which containers are currently running.

03 Core Kafka concepts

These will make full sense once the code runs. Come back here if something is unclear.

Topic
A named channel for messages. Like a named folder. Producers write to it; consumers read from it.
Broker
The Kafka server process. Stores messages and serves them to consumers. In our setup: one broker in Docker.
Producer
Any app that writes messages to a topic. Our Python script is the producer.
Consumer
Any app that reads messages from a topic. Tracks its position with an offset.
Offset
The sequential position of a message in a topic. Kafka saves the last offset per consumer group — no message is missed or duplicated on restart.
Consumer Group
A named group of consumers. Kafka tracks offsets per group. Different groups read the same topic independently.
Partition
Topics can be split into partitions for parallel processing. We use 1 partition today for simplicity.
Serializer
Kafka only transmits bytes. A serializer converts Python dict → JSON → bytes for transport, and back on the consumer side.

      POSTGRES_PASSWORD: pipeline

Line-by-line explanation

depends_on: [zookeeper] — Tells Docker Compose to start Zookeeper before Kafka. Kafka cannot initialize without Zookeeper being ready to accept connections.

KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://localhost:9092 — The address Kafka tells clients to connect to. Since we run everything locally, it's localhost:9092. In a cloud setup this would be the server's public IP or hostname.

KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1 — Kafka normally replicates data across multiple brokers for fault tolerance. Since we have only 1 broker, we set this to 1 to avoid startup warnings.

ports: "5432:5432" — Maps port 5432 inside the container to port 5432 on your laptop. This is how Python can connect to Postgres as if it were running natively.

05 The Producer

The producer simulates an order management system generating purchase events continuously. Each order is serialized as JSON and sent to the orders topic in Kafka.

producer.py 
import json
import time
import random
from datetime import datetime
from faker import Faker
from kafka import KafkaProducer

fake = Faker()

PRODUCTS = [
    {"id": "P001", "name": "Laptop",     "price": 1200.00},
    {"id": "P002", "name": "Headphones", "price":   89.99},
    {"id": "P003", "name": "Keyboard",   "price":   55.00},
    {"id": "P004", "name": "Monitor",    "price":  450.00},
    {"id": "P005", "name": "Mouse",      "price":   35.00},
]

REGIONS = ["North", "South", "East", "West", "Central"]

# Connect to the Kafka broker
producer = KafkaProducer(
    bootstrap_servers='localhost:9092',
    value_serializer=lambda v: json.dumps(v).encode('utf-8')
)

def generate_order():
    product = random.choice(PRODUCTS)
    qty = random.randint(1, 5)
    return {
        "order_id":   fake.uuid4(),
        "customer":   fake.name(),
        "email":      fake.email(),
        "product_id": product["id"],
        "product":    product["name"],
        "qty":        qty,
        "unit_price": product["price"],
        "total":      round(qty * product["price"], 2),
        "region":     random.choice(REGIONS),
        "timestamp":  datetime.utcnow().isoformat(),
    }

print("Producer running. Press Ctrl+C to stop.\n")

while True:
    order = generate_order()
    producer.send('orders', order)
    print(f"  -> sent {order['order_id'][:8]}... | {order['product']} x{order['qty']} | ${order['total']}")
    time.sleep(1)  # one order per second
Key lines explained

KafkaProducer(bootstrap_servers='localhost:9092') — Opens a connection to the broker. The producer uses this initial address to discover all available brokers in the cluster (useful when scaling to multiple brokers).

value_serializer=lambda v: json.dumps(v).encode('utf-8') — Kafka only transports raw bytes. This function automatically converts every Python dict we send into a JSON string, then into bytes. The consumer will do the reverse.

producer.send('orders', order) — Sends the order dict to the topic named 'orders'. If the topic doesn't exist yet, Kafka creates it automatically on the first message.

time.sleep(1) — Throttles to 1 order/second so output is readable in class. Remove this to send as fast as Python can.

📖 What happens if Kafka is down when the producer sends?

By default, producer.send() is asynchronous — it places the message in an internal in-memory buffer and returns immediately. A background thread handles the actual network transmission.

If Kafka is temporarily unavailable, the producer retries automatically (configurable via retries and retry_backoff_ms). Messages wait in the buffer until delivery succeeds or the buffer fills up.

For production systems you'd add producer.flush() to wait for delivery confirmation, and wrap sends in a try/except for KafkaError. We skip that today for clarity.

Try it: Run python producer.py. You should see orders printing to the terminal. Messages are already flowing into Kafka — even before the consumer exists. Kafka stores them durably, waiting.

06 The Consumer (read-only first)

The consumer subscribes to the orders topic and reads every message. We start with a print-only version to observe the flow before wiring up the database.

consumer.py — step 1, read only 
import json
from kafka import KafkaConsumer

consumer = KafkaConsumer(
    'orders',
    bootstrap_servers='localhost:9092',
    auto_offset_reset='earliest',   # start from the very first message
    group_id='sales-pipeline',
    value_deserializer=lambda m: json.loads(m.decode('utf-8'))
)

print("Consumer running. Waiting for messages...\n")

for message in consumer:
    order = message.value
    print(f"  <- received: {order['order_id'][:8]}... | {order['product']} | ${order['total']}")
Key lines explained

auto_offset_reset='earliest' — When this consumer group starts for the first time (no saved offset), it reads from the very beginning of the topic. This means it will catch all orders already sent by the producer. Change to 'latest' to only receive new messages going forward.

group_id='sales-pipeline' — Identifies this consumer as part of the sales-pipeline group. Kafka saves the last read offset per group ID. If you stop and restart the consumer, it picks up exactly where it left off — no messages lost or duplicated.

value_deserializer — The mirror image of the producer's serializer. Automatically converts incoming bytes → JSON string → Python dict on every message received.

for message in consumer: — A blocking loop that waits indefinitely for new messages. message.value is the deserialized dict. You also have access to message.offset, message.partition, and message.timestamp.

📖 What is a Consumer Group and why does it matter?

A consumer group lets multiple consumer instances share the work of reading a topic. Kafka assigns partitions to group members — each partition is processed by exactly one consumer at a time.

In our single-partition setup there's only one slot, so only one consumer reads at a time. If we added a second consumer with the same group_id, it would sit idle — no partition left to assign.

However, if we added a second consumer with a different group_id, it would receive all the same messages independently. This is how you fan out data to multiple downstream systems — analytics, billing, notifications — all reading the same topic without interfering with each other.

Observe: Open two terminals. Run the producer in one, the consumer in the other. Watch messages appear in real time. Stop the consumer, wait 10 seconds, restart it — it will catch up on every message it missed.

07 Writing to PostgreSQL

Now we extend the consumer to persist every message as a row in the database. This is the Load step of ETL — raw data lands in orders_raw exactly as it came from Kafka, no transformation yet.

Step 1 — Create the table

terminal — connect to postgres 
docker exec -it sales-pipeline-postgres-1 psql -U pipeline -d sales
What this command does

docker exec -it — Runs a command inside a running container interactively (-i = keep STDIN open, -t = allocate a TTY), giving you a real terminal session inside the container.

sales-pipeline-postgres-1 — The container name Docker Compose assigns automatically: <project-folder>-<service>-<index>. If your folder name is different, run docker ps to find the exact container name.

psql -U pipeline -d sales — Opens the PostgreSQL CLI connected to database sales as user pipeline, matching the credentials in docker-compose.yml.

postgres prompt — create table 
CREATE TABLE orders_raw (
    id          SERIAL PRIMARY KEY,
    order_id    VARCHAR(36),
    customer    TEXT,
    email       TEXT,
    product_id  VARCHAR(10),
    product     TEXT,
    qty         INTEGER,
    unit_price  NUMERIC(10,2),
    total       NUMERIC(10,2),
    region      VARCHAR(20),
    ts          TIMESTAMP,
    inserted_at TIMESTAMP DEFAULT NOW()
);

\q
Table design notes

SERIAL PRIMARY KEY — Auto-incrementing integer. Each row gets a unique sequential ID automatically — no need to generate one in Python.

NUMERIC(10,2) — Fixed-point decimal: up to 10 digits total, 2 after the decimal point. Always use this for money — never FLOAT, which introduces rounding errors (e.g. $89.99 stored as $89.98999...).

inserted_at TIMESTAMP DEFAULT NOW() — Automatically records when the row landed in Postgres. Combined with ts (the Kafka message timestamp), you can measure pipeline latency: time between order creation and database arrival.

Why a separate orders_raw table? — Best practice is to store data exactly as it arrives, with no transformation. In Class 2 we'll create a cleaned, aggregated table. Keeping raw data intact means you can always re-process it if your business logic changes.

Step 2 — Update consumer.py

consumer.py — final version 
import json
from datetime import datetime
from kafka import KafkaConsumer
import psycopg2

# ── database connection ──────────────────────────────────────────────
conn = psycopg2.connect(
    host="localhost",
    port=5432,
    dbname="sales",
    user="pipeline",
    password="pipeline"
)
cur = conn.cursor()

# ── kafka consumer ───────────────────────────────────────────────────
consumer = KafkaConsumer(
    'orders',
    bootstrap_servers='localhost:9092',
    auto_offset_reset='earliest',
    group_id='sales-pipeline',
    value_deserializer=lambda m: json.loads(m.decode('utf-8'))
)

INSERT_SQL = """
    INSERT INTO orders_raw
        (order_id, customer, email, product_id, product,
         qty, unit_price, total, region, ts)
    VALUES (%s,%s,%s,%s,%s,%s,%s,%s,%s,%s)
"""

print("Consumer + DB running. Writing to PostgreSQL...\n")

for message in consumer:
    o = message.value
    cur.execute(INSERT_SQL, (
        o["order_id"], o["customer"], o["email"],
        o["product_id"], o["product"], o["qty"],
        o["unit_price"], o["total"], o["region"],
        datetime.fromisoformat(o["timestamp"])
    ))
    conn.commit()
    print(f"  saved {o['order_id'][:8]}... -> {o['product']} x{o['qty']}")
Key lines explained

psycopg2.connect(...) — Opens a persistent TCP connection to PostgreSQL. We open it once before the loop — creating a new connection per message would be extremely slow (TCP handshake + auth on every row).

conn.cursor() — A cursor is the object that actually executes SQL statements. Think of it as a "session handle" within the connection. One connection can have multiple cursors.

INSERT_SQL with %s placeholders — Parameterized queries prevent SQL injection and handle proper quoting automatically. Never use f-strings or string concatenation to build SQL — this is both a security and correctness issue.

conn.commit() — Flushes the transaction to disk. Without this, the INSERT exists only in memory and is lost if Python crashes. For high-throughput production systems you'd batch multiple inserts per commit.

datetime.fromisoformat(o["timestamp"]) — Converts the ISO 8601 string from Kafka ("2024-01-15T10:30:00") back into a Python datetime object, which psycopg2 stores correctly as a PostgreSQL TIMESTAMP.

Step 3 — Verify data in Postgres

postgres prompt — verification queries 
-- How many orders have arrived?
SELECT COUNT(*) FROM orders_raw;

-- Inspect the latest 5 orders
SELECT order_id, product, qty, total, region, ts
FROM orders_raw
ORDER BY ts DESC
LIMIT 5;

-- Quick summary by product
SELECT product, COUNT(*) AS orders, SUM(total) AS revenue
FROM orders_raw
GROUP BY product
ORDER BY revenue DESC;
