02 ETL vs ELT

Both patterns move data from source to destination — the difference is where transformation happens.

ETL — Extract, Transform, Load

Transform first, then load. Data is cleaned and shaped before it enters the target system.

Best when the destination has limited compute (e.g. a small Postgres), or when data quality must be enforced before storage.

Our pipeline today: orders_raw → Python transformer → star schema in Postgres.

ELT — Extract, Load, Transform

Load raw first, transform inside. Raw data lands in the warehouse and SQL/dbt handles transformation.

Best when the destination is powerful (BigQuery, Snowflake, Redshift) and transformation can run as SQL at scale.

Cloud version of this pipeline: Kafka → Pub/Sub → BigQuery raw → dbt → dashboard.

📖 Where does Kafka fit in both patterns?

Kafka is the ingestion and buffering layer — it sits between the source and the transformation step in both ETL and ELT.

In ETL: Producer → Kafka → Consumer (transforms) → clean DB. Transformation lives in the consumer.

In ELT: Producer → Kafka → Consumer (raw load) → warehouse → SQL transforms. Transformation moves to the warehouse.

Kafka's role doesn't change: it decouples the producer from whatever comes after. What changes is whether the consumer transforms before or after storing.

📖 What is a Star Schema and why does it matter for dashboards?

A star schema organizes data into one central fact table (the measurements — in our case, each order) surrounded by dimension tables (the context — products, regions, time).

It's called a star because the diagram looks like one: fact table in the center, dimensions radiating outward.

Why it matters for dashboards: BI tools like Looker Studio, Power BI, and Tableau are optimized for this shape. They can generate charts and filters directly from the relationships between fact and dimension tables without complex SQL.

Normalized (3NF) vs Star: A normalized schema minimizes redundancy — great for transactional systems. A star schema accepts controlled redundancy in exchange for simpler, faster analytical queries.

03 The Transformer

The transformer is a standalone Python script — not a Kafka consumer. It reads from orders_raw, applies cleaning and enrichment logic, and loads the results into the star schema tables.

This is a batch transformation: run it once, or schedule it with cron or Airflow. In Class 1 we did real-time ingestion; now we do scheduled transformation — a common real-world combination.

    qty         INTEGER,
    unit_price  NUMERIC(10,2),

    revenue_cat TEXT,   -- 'low' | 'medium' | 'high'
    inserted_at TIMESTAMP DEFAULT NOW()
);

Schema design decisions

dim_product.price_tier — A derived column: we categorize products by price range during transformation. "Laptop" → premium, "Mouse" → budget. This is a classic transformation that would be expensive to do at query time on every dashboard load.

dim_time — Breaking timestamps into year/month/day/hour/weekday columns makes time-based filtering and grouping trivially fast in BI tools. "Show me sales by hour of day" becomes a simple GROUP BY hour instead of EXTRACT(HOUR FROM ts) on millions of rows.

fact_orders.revenue_cat — Categorizes each order as low/medium/high revenue. Useful for segmentation charts without needing CASE statements in every query.

Foreign keys — The REFERENCES constraints enforce referential integrity. A fact row cannot reference a product that doesn't exist in dim_product. This prevents orphan records from entering the analytical layer.

Step 2 — transformer.py

transformer.py 
"""
transformer.py
Reads from orders_raw, transforms, and loads into the star schema.
Run after consumer.py has been collecting data for a while.
"""

import psycopg2
import psycopg2.extras

# ── connection ────────────────────────────────────────────────────────────────
conn = psycopg2.connect(
    host="localhost", port=5432,
    dbname="sales", user="pipeline", password="pipeline"
)
cur = conn.cursor(cursor_factory=psycopg2.extras.RealDictCursor)

# ── helpers ───────────────────────────────────────────────────────────────────
def price_tier(price):
    if price <= 60:    return 'budget'
    if price <= 500:   return 'mid-range'
    return 'premium'

def revenue_cat(total):
    if total < 100:    return 'low'
    if total < 500:    return 'medium'
    return 'high'

WEEKDAYS = ['Monday','Tuesday','Wednesday','Thursday','Friday','Saturday','Sunday']

# ── extract: read all raw orders ──────────────────────────────────────────────
print("Extracting from orders_raw...")
cur.execute("SELECT * FROM orders_raw ORDER BY id")
raw_orders = cur.fetchall()
print(f"  {len(raw_orders)} rows found")

# ── transform + load ──────────────────────────────────────────────────────────
print("Transforming and loading...")

loaded = 0
for row in raw_orders:

    # 1. populate dim_product (upsert)
    cur2 = conn.cursor()
    cur2.execute("""
        INSERT INTO dim_product (product_id, product_name, price_tier)
        VALUES (%s, %s, %s)
        ON CONFLICT (product_id) DO NOTHING
    """, (row['product_id'], row['product'], price_tier(float(row['unit_price']))))

    # 2. populate dim_region (upsert)
    cur2.execute("""
        INSERT INTO dim_region (region_name)
        VALUES (%s)
        ON CONFLICT (region_name) DO NOTHING
    """, (row['region'],))

    # 3. get region_id
    cur2.execute("SELECT region_id FROM dim_region WHERE region_name = %s", (row['region'],))
    region_id = cur2.fetchone()[0]

    # 4. populate dim_time (upsert on ts)
    ts = row['ts']
    if ts:
        cur2.execute("""
            INSERT INTO dim_time (ts, year, month, day, hour, weekday)
            VALUES (%s, %s, %s, %s, %s, %s)
            ON CONFLICT (ts) DO NOTHING
        """, (
            ts,
            ts.year, ts.month, ts.day, ts.hour,
            WEEKDAYS[ts.weekday()]
        ))
        cur2.execute("SELECT time_id FROM dim_time WHERE ts = %s", (ts,))
        time_id = cur2.fetchone()[0]
    else:
        time_id = None

    # 5. insert into fact_orders (skip duplicates by order_id)
    cur2.execute("SELECT 1 FROM fact_orders WHERE order_id = %s", (row['order_id'],))
    if not cur2.fetchone():
        cur2.execute("""
            INSERT INTO fact_orders
                (order_id, product_id, region_id, time_id,
                 customer, qty, unit_price, total, revenue_cat)
            VALUES (%s,%s,%s,%s,%s,%s,%s,%s,%s)
        """, (
            row['order_id'], row['product_id'], region_id, time_id,
            row['customer'], row['qty'],
            row['unit_price'], row['total'],
            revenue_cat(float(row['total']))
        ))
        loaded += 1

    conn.commit()

print(f"  {loaded} new rows loaded into fact_orders")
print("Done.")
Key patterns explained

ON CONFLICT (product_id) DO NOTHING — An upsert pattern. If the product already exists in dim_product, skip it. This makes the transformer idempotent — safe to run multiple times without creating duplicate dimension rows.

price_tier() and revenue_cat() — These are business rules encoded as Python functions. In a real company, these thresholds would come from a config file or database table so analysts can change them without touching code.

Duplicate check on order_id — Before inserting into fact_orders, we verify the order hasn't been loaded before. This prevents double-counting if the transformer is run multiple times.

ts.weekday() — Python's datetime.weekday() returns 0=Monday, 6=Sunday. We map this to readable day names for dashboard labels.

Run it: With the consumer running (data accumulating in orders_raw), open a third terminal and run python transformer.py. Check the output — it should report how many rows were loaded. Run it again — it should report 0 new rows (idempotent).
📖 What is idempotency and why does it matter in ETL?

An operation is idempotent if running it multiple times produces the same result as running it once. For ETL jobs, this is critical.

Real pipelines fail — network issues, database timeouts, out-of-memory errors. When a job fails halfway through, you need to be able to re-run it safely without corrupting or duplicating data.

Our transformer is idempotent because: (a) dimension upserts use ON CONFLICT DO NOTHING, and (b) fact inserts check for existing order_id before inserting. No matter how many times you run it, the final state is always correct.

postgres prompt — verify star schema 
-- Row counts in all tables
SELECT 'dim_product'  AS tbl, COUNT(*) FROM dim_product
UNION ALL
SELECT 'dim_region',  COUNT(*) FROM dim_region
UNION ALL
SELECT 'dim_time',    COUNT(*) FROM dim_time
UNION ALL
SELECT 'fact_orders', COUNT(*) FROM fact_orders;

-- Revenue by product (joining fact + dimension)
SELECT
    p.product_name,
    p.price_tier,
    COUNT(*)        AS orders,
    SUM(f.qty)      AS units,
    SUM(f.total)    AS revenue
FROM fact_orders f
JOIN dim_product p USING (product_id)
GROUP BY p.product_name, p.price_tier
ORDER BY revenue DESC;

-- Orders by region
SELECT
    r.region_name,
    COUNT(*)     AS orders,
    SUM(f.total) AS revenue
FROM fact_orders f
JOIN dim_region r USING (region_id)
GROUP BY r.region_name
ORDER BY revenue DESC;

-- Orders by hour of day
SELECT
    t.hour,
    t.weekday,
    COUNT(*) AS orders
FROM fact_orders f
JOIN dim_time t USING (time_id)
GROUP BY t.hour, t.weekday
ORDER BY t.hour;

05 Connecting Looker Studio

Looker Studio (formerly Google Data Studio) is a free, browser-based BI tool that connects directly to PostgreSQL. We'll build three charts from the star schema we just created.

Prerequisite: Your PostgreSQL must be reachable from Looker Studio's servers. Since ours runs on localhost, we need to expose it. The simplest option for class: use ngrok to create a temporary public tunnel. Run ngrok tcp 5432 and use the generated hostname in the connector.

Option A — ngrok tunnel (quick, for class)

terminal 
# Install ngrok if needed: https://ngrok.com/download
# Then:
ngrok tcp 5432

# Output will look like:
# Forwarding  tcp://0.tcp.ngrok.io:12345 -> localhost:5432
# Use "0.tcp.ngrok.io" as host and "12345" as port in Looker Studio
📖 Option B — run everything on a cloud VM (production approach)

For a real production setup, the entire stack (Kafka, Postgres, Python scripts) runs on a cloud VM with a public IP — for example a Google Cloud Compute Engine instance or an AWS EC2 instance.

You would open port 5432 in the firewall rules, and Looker Studio connects directly to the public IP without ngrok.

This is also how you'd connect Power BI Desktop: install the PostgreSQL ODBC driver, then use "Get Data → PostgreSQL database" and enter the server's public IP.

Add the PostgreSQL data source in Looker Studio

  1. Go to lookerstudio.google.com → Create → Report.
  2. Search for the PostgreSQL connector and select it.
  3. Enter the connection details using the block below, then click Authenticate.
  4. Select table fact_orders as the primary source.
  5. Click Add to report.
